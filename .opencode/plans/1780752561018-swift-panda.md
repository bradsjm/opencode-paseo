# IMPLEMENTATION SPEC

## 1. Objective

Migrate the repository ESLint TypeScript preset from `typescript-eslint` `recommended` to `recommendedTypeChecked` and complete the required cleanup for newly enforced type-aware rules. Definition of done: `eslint.config.js` uses `recommendedTypeChecked`, the cited unsafe patterns in `index.ts`, `lib/config.ts`, and `lib/transport/client.ts` are resolved without rule suppressions, and the repo passes lint, typecheck, and unit tests.

## 2. Scope

- In scope:
  - Update `eslint.config.js` to apply `tseslint.configs.recommendedTypeChecked` to TypeScript files.
  - Keep type-aware parser project wiring for `index.ts`, `lib/**/*.ts`, `tsup.config.ts`, and `tests/**/*.ts`.
  - Fix type-aware lint findings caused by the migration in production code and tests when they are surfaced by `pnpm lint`.
  - Clean up known hotspots at `index.ts:79`, `index.ts:126`, `lib/config.ts:130`, `lib/config.ts:344`, and `lib/transport/client.ts:393/403`.
- Out of scope:
  - Do not redesign plugin startup, config loading, daemon transport, or permission workflows.
  - Do not change daemon protocol semantics or tool argument contracts.
  - Do not replace Zod/jsonc-parser config validation.
  - Do not add dependency upgrades unrelated to ESLint preset migration.
- Assumptions:
  - The implementer should only fix rule findings introduced or exposed by `recommendedTypeChecked`.
  - Preserve existing runtime behavior unless a current unsafe access must be narrowed to keep the same behavior safely.

## 3. Relevant context

- `eslint.config.js` currently uses `extends: [...tseslint.configs.recommended]` for `**/*.ts`.
- `eslint.config.js` already has `parserOptions.project: ["./tsconfig.json"]` for `index.ts`, `lib/**/*.ts`, and `tsup.config.ts`.
- `eslint.config.js` already has `parserOptions.project: ["./tsconfig.tests.json"]` for `tests/**/*.ts` and disables `@typescript-eslint/no-floating-promises` there.
- `package.json` scripts: `pnpm lint` runs `eslint . && prettier --check .`; `pnpm typecheck` runs both main and test TypeScript projects; `pnpm test` runs Node tests through `tsx`.
- `index.ts` is the plugin assembly layer. It catches daemon connect and dispose errors with `catch (err: any)` and logs `err.message`.
- `lib/config.ts` loads JSON/JSONC files as `unknown | null`, validates with `configLayerSchema.safeParse`, and currently skips empty data with `if (!result.data) continue`.
- `lib/transport/client.ts` is the daemon protocol boundary. `mapDaemonEvent` currently casts permission request and resolution payloads with `as unknown as Record<string, unknown>`.
- `lib/transport/types.ts` defines normalized permission event payloads as `request: Record<string, unknown>` and `resolution: Record<string, unknown>`.
- `lib/hooks/daemon-events.ts` expects `permission.requested.payload.request` to contain an optional string `id` used by downstream pending-permission state.

## 4. Constraints

- Language/framework: ESM TypeScript on Node >=18, using ESLint 9 flat config and `typescript-eslint` 8.
- Style: keep existing Prettier formatting and repository import style.
- Architecture: keep `index.ts` as assembly only; keep daemon payload normalization inside `lib/transport/client.ts`; keep state mapping data-oriented.
- Safety: do not use `any` or unsafe double assertions to silence type-aware rules.
- Rule handling: do not disable `recommendedTypeChecked` rules globally or locally unless a confirmed false positive has no small safe rewrite.
- Config behavior: preserve localhost-only daemon host validation and existing global config auto-creation behavior.
- Dependencies: do not add new dependencies. Existing `typescript-eslint` already exposes `recommendedTypeChecked`.
- Compatibility: do not add fallback compatibility shims or legacy branches.

## 5. Implementation steps

1. Update `eslint.config.js` by replacing `tseslint.configs.recommended` with `tseslint.configs.recommendedTypeChecked` in the `**/*.ts` block. Keep the existing project-specific parser options and custom rules unchanged initially.
2. Run `pnpm lint` to capture all type-aware ESLint failures after the preset switch. Treat the output as the cleanup checklist.
3. In `index.ts`, replace both `catch (err: any)` blocks with `catch (err: unknown)`. Add a small local helper or inline expression that logs `err.message` only when `err instanceof Error`; otherwise log `String(err)`.
4. In `lib/config.ts`, keep `ConfigLoadResult.data` as `unknown | null`, but replace truthiness checks such as `if (!result.data) continue` with explicit `result.data === null` checks so valid falsy parsed values still flow to Zod validation and invalid shapes produce warnings.
5. In `lib/transport/client.ts`, reuse or extend the existing `asRecord(value: unknown): Record<string, unknown> | null` helper for permission request and resolution payloads. Map non-record request/resolution values to `{}` or omit them only if the normalized type is changed deliberately; recommended default is `{}` to preserve the required `Record<string, unknown>` shape.
6. In `lib/transport/client.ts`, derive `permissionId` from the normalized request record with `typeof request.id === "string" ? request.id : undefined` instead of optional-access on an opaque upstream value.
7. Run `pnpm lint` again and fix any additional `recommendedTypeChecked` findings with local type guards, explicit unknown narrowing, or typed helper functions. Keep fixes inside the module where the unsafe boundary exists.
8. Run `pnpm typecheck` and `pnpm test`. If permission or transport behavior changes beyond local narrowing, also run `pnpm test:integration`.
