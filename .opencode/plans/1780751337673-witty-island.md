# IMPLEMENTATION SPEC

## 1. Objective

Add one plugin-wide OpenCode `tool.definition` hook that converts every registered Paseo tool's Zod `args` shape into a correct LLM/API-facing JSON Schema and assigns it to `output.jsonSchema`, so optional tool arguments remain optional when the LLM or API wrapper calls the tool. The definition of done is that every `paseo_*` tool advertises required fields only when its Zod arg is required, complex shapes such as nested objects, records, arrays, defaults, and nullable fields are preserved, and the package declares a minimum OpenCode plugin runtime that supports `tool.definition.output.jsonSchema`.

## 2. Scope

- In scope:
  - Centralize all registered `paseo_*` tool definitions into a reusable `tools` object in `index.ts`.
  - Add a plugin-wide `"tool.definition"` hook for Paseo tools only.
  - Generate JSON Schema from each tool definition's existing Zod `args` object.
  - Preserve optionality, nested object required arrays, records, arrays, enums, descriptions, defaults, and nullable fields.
  - Raise the package's minimum `@opencode-ai/plugin` compatibility floor to a version that exposes `output.jsonSchema` on `tool.definition`.
  - Add focused tests for schema conversion and hook behavior.
- Out of scope:
  - Do not modify OpenCode itself.
  - Do not change daemon transport contracts.
  - Do not rewrite individual tool runtime behavior unless a test exposes a schema-specific mismatch.
  - Do not encode conditional runtime requirements as top-level JSON Schema `required` fields.
  - Do not add a new external schema-conversion dependency unless local Zod conversion is proven insufficient.
- Assumptions:
  - The installed `@opencode-ai/plugin` API is authoritative.
  - The lockfile currently resolves `@opencode-ai/plugin@1.15.13`, which includes `tool.definition` with `jsonSchema` support.
  - The implementer must not expand this into a broader tool redesign.

## 3. Relevant context

- `index.ts` is the plugin assembly layer. It currently returns `tool: { ... }` inline with 45 registered `paseo_*` tools.
- `lib/hooks.ts` owns OpenCode hook entrypoints. New hook wiring should be exported from there or from a nested file re-exported there.
- `lib/tools/` owns tool definitions and argument contracts. Existing tools already define optional args with Zod, for example `lib/tools/terminal.ts` marks `start`, `end`, `scrollback`, and `stripAnsi` optional for `paseo_terminal_capture`.
- OpenCode's current plugin type exposes `"tool.definition"?: (input, output) => Promise<void>` where `output` has `description`, `parameters`, and `jsonSchema`.
- Upstream OpenCode prefers `tool.jsonSchema` over deriving JSON Schema from internal parameters when available.
- Current behavior observed from the API wrapper shows optional fields being treated as required, for example `paseo_terminal_capture` requiring `start`, `end`, `scrollback`, and `stripAnsi` even though the Zod schema and runtime handling make them optional.
- Tool schema edge cases that must be preserved:
  - `paseo_worker_update.settings.model` and `settings.thinkingOptionId` are nullable optional fields.
  - `paseo_worker_create.labels`, `paseo_worker_run.labels`, and `paseo_worker_update.labels` are string-to-string records.
  - `paseo_worker_update.settings.features` is a string-to-unknown record.
  - `paseo_terminal_send_lines.lines` and `paseo_worker_wait.workerIds` have array minimum length constraints.
  - `paseo_schedule_create`, `paseo_schedule_update`, and `paseo_loop_run` have conditional/runtime validation that should stay in execute logic.

## 4. Constraints

- Use TypeScript ESM patterns already used in the repo.
- Keep `index.ts` as the assembly layer only: create the `tools` map there, then pass it to a hook factory.
- Keep hook-specific logic under `lib/hooks.ts` or `lib/hooks/tool-definition.ts`.
- Keep pure schema conversion logic small and testable. Prefer `tool.schema.toJSONSchema(tool.schema.object(args), { io: "input" })` using the same Zod instance exposed by `@opencode-ai/plugin/tool`.
- Do not make optional fields required to satisfy provider strictness.
- Do not add default values to tool schemas unless they already exist in Zod metadata or are naturally emitted by Zod.
- Do not duplicate per-tool JSON Schema by hand.
- New runtime dependency is not allowed by default. Use the existing `@opencode-ai/plugin`/Zod API.
- Development/peer dependency version updates are allowed because the feature requires a newer OpenCode plugin API.
- Raise `peerDependencies["@opencode-ai/plugin"]` to at least `>=1.15.13` and align `devDependencies["@opencode-ai/plugin"]` with that floor.
- Update lockfiles only as a consequence of dependency metadata alignment during implementation.

## 5. Implementation steps

1. Inspect the installed `@opencode-ai/plugin` types and confirm that `Hooks["tool.definition"]` includes mutable `output.jsonSchema`. Use `@opencode-ai/plugin@1.15.13` as the minimum supported floor if confirmed.
2. In `index.ts`, replace the inline `tool: { ... }` literal with a local `const tools = { ... }` containing the exact same 45 `paseo_*` tool definitions and no behavior changes.
3. Add a hook factory, preferably `createToolDefinitionHandler` in `lib/hooks/tool-definition.ts`, and re-export it from `lib/hooks.ts`. The factory should accept the `tools` map and optionally the logger.
4. Add a pure converter helper, preferably `createToolArgsJsonSchema(args)` in `lib/tools/json-schema.ts` or colocated under `lib/hooks/tool-definition.ts` if it stays small. Build `tool.schema.object(args)` and pass it to `tool.schema.toJSONSchema(..., { io: "input" })`. Normalize only compatibility details such as root object validation and `$defs` to `definitions` if needed.
5. Implement the `"tool.definition"` hook so it no-ops for non-`paseo_*` IDs or unknown IDs, looks up the registered tool by `toolID`, generates JSON Schema from `def.args`, and assigns the result to `output.jsonSchema`. Do not throw for unrelated tools.
6. Wire the hook in `index.ts` next to the existing `event` and `config` hooks: `"tool.definition": createToolDefinitionHandler(tools, logger)`. Keep `tool: tools` unchanged.
7. Update `package.json` so `peerDependencies["@opencode-ai/plugin"]` is `>=1.15.13` and `devDependencies["@opencode-ai/plugin"]` is aligned to `^1.15.13` or the repo's chosen exact compatible range. Update any local `.opencode/package.json` plugin-runtime metadata if it pins the older version.
8. Add unit tests for the converter and hook. Cover `paseo_terminal_capture`, `paseo_worker_create`, `paseo_worker_update`, `paseo_schedule_create`, and one no-arg tool such as `paseo_status`.
9. Run verification commands and fix only issues related to schema conversion, hook typing, version metadata, or tests.

## 6. Acceptance checks

- Functional checks:
  - `paseo_terminal_capture` JSON Schema has `required: ["terminalId"]` only.
  - `start`, `end`, `scrollback`, and `stripAnsi` remain in `properties` but not in `required`.
  - All-optional tools such as `paseo_worker_create` emit no required fields or an absent/empty `required` array.
  - No-arg tools emit a valid object schema with no required fields.
- Edge-case checks:
  - `paseo_worker_update` preserves nested `settings` as optional and preserves nullable `model` and `thinkingOptionId`.
  - Record fields emit `additionalProperties` correctly for string records and permissive unknown records.
  - Array fields preserve `items` and minimum length constraints where present.
  - Runtime-only conditional requirements remain described and enforced in existing execute logic, not promoted into top-level schema required fields.
- Regression checks:
  - Existing tool execution tests still pass without requiring callers to supply omitted optional args.
  - Unknown or non-Paseo tools passed through the hook are not modified.
  - Plugin startup still degrades gracefully when the daemon is unavailable.
- Validation commands:
  - `pnpm typecheck`
  - `pnpm test`
  - `pnpm lint`
  - `pnpm test:integration` when the `opencode` CLI is available on `PATH`
  - Targeted development command if needed: `node --import tsx --test tests/hooks.test.ts tests/tool-json-schema.test.ts`
